### 1.1 Hard constraints (non-negotiable, from the owner)

1. All work happens in the personal fork (`fork` = `mmackelprang/claude-mem`).
2. **Nothing is pushed to upstream.** `origin`'s push URL is already set to `DISABLED_UPSTREAM_DO_NOT_PUSH` — verified. Do not re-enable it. Do not propose contributing anything upstream, even where our code is better.
3. Keep **all** of our features. Lose no WS2 work.
4. Branch-per-change, merge via PR; PRs target `fork/main`.

Constraint 2 is load-bearing for the decision in §2 — see the rationale.

### 1.2 Cleared risks (independently re-verified for this ADR)

- **No migration collision.** `SERVER_POSTGRES_SCHEMA_VERSION`: upstream `1` (`origin/main:src/storage/postgres/schema.ts:6`), fork `3` (`fork/main:src/storage/postgres/schema.ts:5`). Upstream added zero DDL. Our v2/v3 are uncontested.

- **ChromaSync's −245 is a DRY collapse + DI**, not a restructure. All Phase 4 anchors survive (`formatObservationDocs` 3/3, `addDocuments`, `queryChroma`, `deduplicateQueryResults`). The `backfillAllProjects(this.dbManager.getSessionStore())` call site is byte-identical (`fork:src/services/worker-service.ts:603` ≡ `upstream:601`).

## 2. Decision

**Perform a single `git merge origin/main` into a branch cut from `fork/main`, resolve the 18 conflicts once, and open one PR against `fork/main`.** Do not rebase. Do not re-apply our work as fresh PRs.

### 2.1 Options considered

| Option | Verdict | Rationale |
|---|---|---|
| **Merge** (chosen) | ✅ | One resolution pass. Preserves both histories and records the upstream lineage in the DAG, so the *next* merge's base is `f5633c1f` rather than `cbdce2d6` — this is the compounding benefit. |
| **Rebase** `fork/main` onto `origin/main` | ❌ | Replays 55 commits across 91, re-fighting the same conflicts repeatedly. Our exposure is concentrated in files upstream rewrote wholesale (`ServerV1PostgresRoutes` 137 ours vs 672 theirs; `ChromaSync` 98 vs 346; `mcp-server` 45 vs 489) — each replayed commit re-hits them. It also rewrites published history that the pilot NAS deploys from. |
| **Re-apply as fresh PRs** | ❌ | Discards working, reviewed history. Justified only if the anchors were gone — they aren't (§1.2). No gain. |

### 2.2 Why the "nothing goes upstream" constraint decides this

Rebase's principal benefit is a clean, linear, upstream-presentable history. **Constraint 2 makes that benefit worth exactly zero.** We will never open a PR against `thedotmack/claude-mem`, so nobody outside this fork ever reads our commit topology. What we need instead is a durable record of *which upstream commit we last reconciled against* — which is precisely what a merge commit encodes and a rebase erases. The constraint doesn't merely permit the merge; it actively selects it.

Corollary: this merge is **strictly one-directional and repeatable**. Treat `origin/main` as a read-only vendor feed. This ADR's rules are written to be re-runnable at the next upstream sync.

#### 4.3.2 `src/storage/postgres/schema.ts` — ⚠️ P0 correctness hazard

Adopt upstream's structure: `bootstrapServerPostgresSchema` (`upstream:28`) delegating to `applyPhase1Migration(client)` (`upstream:51`). Keep our `SERVER_POSTGRES_SCHEMA_VERSION = 3`. Add `applyPhase2Migration` / `applyPhase3Migration` in the same style. Our tail DDL (`actor_id`, `api_key_id`, `visibility`) has zero upstream overlap — clean append.

**The hazard.** Upstream's `applyPhase1Migration` records its migration row using the *constant*:

```ts
[SERVER_POSTGRES_SCHEMA_VERSION, 'phase 1 postgres observation storage foundation']  // upstream:59
```

Upstream's constant is `1`, so this is correct *for them*. **Our constant is `3`** (= "latest"), so adopting their function verbatim stamps the phase-1 row as **version 3**.

Our fork deliberately uses **literals** (`fork:47` → `[1, …]`, `fork:55` → `[2, …]`, `fork:76` → `[3, …]`) precisely because the constant now means "latest", not "this migration".

Consequences if the constant leaks into `applyPhase1Migration`:
1. Our Phase-3 guard — `SELECT 1 FROM server_beta_schema_migrations WHERE version = $1` with `[3]` (`fork:65-67`) — finds the bogus row, so `rowCount !== 0`.
2. The one-time `UPDATE observations SET visibility = 'private'` backfill is **skipped forever** on every fresh database.
3. Pre-visibility rows keep the `'team'` column default → **history that predates any sharing intent becomes team-visible**. This is a privacy **fail-open**.
4. `ON CONFLICT (version) DO NOTHING` then silently suppresses the real v3 row, leaving a wrong description in `server_beta_schema_migrations`.

**It fails silently — no compile error, no test failure, and the existing pilot is unaffected** (its v3 row already exists), so this will not surface until a fresh DB or a teammate onboards.

**Rule: inside `applyPhase1Migration`, replace `SERVER_POSTGRES_SCHEMA_VERSION` with the literal `1`.** Each `applyPhaseNMigration` records its own literal `N`. The constant is reserved for "latest schema version" and must appear in no migration row.

Upstream moved `pg`, `zod`, and `@modelcontextprotocol/sdk` from `dependencies` → `devDependencies`. ## 5. Phasing

### Phase 0 — Pre-merge alignment on `fork/main` (2 PRs; recommended, not required)

Both PRs are pure fork-side alignment with upstream's already-made decisions, independently verifiable with **no upstream code in the tree**. They shrink the merge PR and convert two silent hazards into small reviewed changes.

  > **Ordering constraint:** the SDK teardown and the OVERRIDE-C removal **must ship together**. `tsup`'s `build:sdk` entry point is `src/sdk/index.ts`; deleting the barrel without removing the tsup/check-sdk-bundle chain breaks the build. This is why it's one PR, not two.
- **PR #B — `chore/pre-merge-align-audit-rename`.** `auditRead` → `auditWrite` at 9 call sites; delete the alias (`fork:1225`). Provably safe (§4.3.4). Converges both sides on the same text, so those lines auto-merge instead of conflicting.

**Gate for each:** `npm run build-and-sync` green + full unit suite green.

**Honest trade-off:** PR #A carries most of the value (removes the modify/delete conflict and the OVERRIDE-C tangle). PR #B is noise reduction only — `ServerV1PostgresRoutes.ts` still conflicts regardless, because our +137 meets upstream's ±672. If the owner prefers a single unit of work, fold both into Phase 1 as its first two commits; the resolution rules are unchanged.

### Phase 1 — The merge (1 PR — necessarily atomic)

A merge commit cannot be split across PRs, and a partially-resolved tree doesn't build. **This is one PR.** Sequence the commits inside it so each is reviewable and the tree builds at commits 2→4:

1. **`git merge origin/main`** — resolve the **source/test** conflicts only (§4.2–§4.4). Do **not** touch `plugin/scripts/*.cjs`. *Tree does not build yet.*

3. **Test fallout** — §6. *Suite green.*

## 7. Rollback plan & risk register

### 7.1 Rollback

- **Pre-merge:** delete the branch. Zero blast radius.
- **Post-merge, pre-deploy:** `fork/main` is a merge commit → `git revert -m 1 <merge-sha>` restores the pre-merge tree. Keep the branch so the resolution work isn't lost.
- **Post-deploy:** roll the pilot back to the current `13.9.2`-based bundle (§7.2). **The schema needs no rollback** — upstream added zero DDL, so v3 is a superset of anything the merged code expects. This is what makes the redeploy comparatively safe: **the bundle changes substantially; the database contract does not.**
- **The DB is the one-way door.** Nothing in this merge migrates data. If §4.3.2 is botched the damage is a *missing* backfill on fresh DBs, not corruption of the pilot — recoverable by fixing the guard and re-running.

### 7.2 Pilot redeploy sequencing

The pilot NAS runs a `13.9.2`-based fork build: Postgres + Redis + `server-service` via docker-compose (`docs/ops/2026-07-03-nas-tailscale-pilot-runbook.md:37`).

1. Merge to `fork/main`; confirm build + suite green.
2. **Regenerate and commit `plugin/scripts/server-service.cjs`** — the §4.1 gate. *Precedent: runbook:49 — a source-only merge already shipped a stale bundle at the wrong schema version once.*
3. Snapshot/back up the pilot Postgres volume before touching the container.

## 9. Consequences

**Good**
- One conflict-resolution pass instead of 55. Both histories preserved.
- The DAG records `f5633c1f` as reconciled → the **next** upstream sync bases on it, not on `cbdce2d6`. The cost of this merge is paid once.
- We inherit upstream's OVERRIDE-C dissolution, the `generateAndPersist` extraction, the ChromaSync DRY collapse, and the atomic-JSON/logger improvements — deleting fork-maintained complexity.
- Establishes a repeatable one-directional vendor-merge procedure.

**Bad / accepted**
- The merge PR is unavoidably large and partly generated. Mitigated by Phase 0 + commit sequencing, not eliminated.
- We permanently carry fork-only divergence (packaging `.cjs` emit, `/v1/search` routing, `memories:*` minting, the tag-stripping fail-safe, the whole WS2 surface). Every future sync re-fights the same regions. **This is the deliberate price of constraint 2** — the alternative (upstreaming our fixes so they stop being divergence) is explicitly excluded.
- Divergence is now **asymmetric and permanent**: upstream will keep refactoring regions we've modified, so friction compounds. Syncing more often makes each merge cheaper.
- Two P0 hazards (R1, R2) are invisible to the compiler; only one is test-caught.

**Neutral**
- `pg`/`zod`/MCP-SDK in `devDependencies` is a topology we now inherit rather than choose — safe today because everything is bundled, and a trap for any future unbundled entry point.
